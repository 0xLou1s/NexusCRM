# @workspace/db

Drizzle schema, migrations and the database client factory. Owned by the
backend: `api` and `zalo-worker` both consume it, the frontend never does.

## Layout

- `src/schema/` — one file per table group, re-exported from `src/schema/index.ts`
- `src/migrations/` — SQL generated by drizzle-kit; committed, never edited after it has been applied
- `src/client.ts` — `createDatabase()`, a factory so each process owns its pool

## Environment

| Variable              | Used by          | Supabase source                     |
| --------------------- | ---------------- | ----------------------------------- |
| `DATABASE_URL`        | the running apps | Transaction pooler (port 6543)      |
| `DIRECT_DATABASE_URL` | drizzle-kit only | Session pooler / direct (port 5432) |

Both live in the repository-root `.env`. The transaction pooler cannot hold
prepared statements (hence `prepare: false` in the client) and does not support
the advisory locks migrations need — so drizzle-kit uses the direct URL when it
is present, and falls back to `DATABASE_URL` otherwise.

## Scripts

```sh
pnpm --filter @workspace/db db:generate   # schema change -> new SQL migration
pnpm --filter @workspace/db db:migrate    # apply pending migrations
pnpm --filter @workspace/db db:studio     # browse the database
```

Changing a table means: edit the schema, run `db:generate`, review the emitted
SQL, then `db:migrate`. Never hand-write a migration that drizzle-kit could have
generated, and never edit a migration that has already been applied.
